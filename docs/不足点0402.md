❌ 文档建议但项目完全缺失的部分
1. 端到端链路测试（P0）
文档明确要求 P0 测试："给一条采集内容，最终能不能生成合规文案"。项目测试文件有 test_api.py、test_main.py 等，但没有一个从采集→清洗→标签→知识库→检索→生成→合规的完整链路测试。这是最急迫要补的。
2. 生成输出结构化
文档要求输出必须固定包含：标题、开头钩子、正文、行动引导、风险点、替代版本 1/2/3。当前 mvp_generate_core_service.py 生成多版本，但输出结构没有强制包含风险点说明和替代表达。需要升级 generate_schema.py 和 Prompt。
3. 知识库三层分离
文档要求明确分出：原始内容池 → 优质素材池 → 结构化知识库。当前有 MvpInboxItem（收件箱）→ MvpMaterialItem（素材）→ MvpKnowledgeItem（知识），数据模型已具备三层，但"优质仿写池"的概念在筛选流程中未显式落地——quality_screening_service 计算了分数，但没有看到明确的"分数达标自动进入仿写池"的标记逻辑。
4. 助贷专用客户字段
文档要求 Customer 补充：贷款需求类型、期望金额、职业身份、社保/公积金、房/车/营业执照、征信、负债区间、可匹配产品类型。当前 LeadProfile 覆盖了部分（has_house/has_car/has_provident_fund/credit_status），但 Customer 模型本身缺乏贷款业务字段（如期望金额、职业身份、负债区间、可匹配产品）。
5. 审计日志持久化
core/audit.py 只是写日志文件。文档要求"谁改了文案、谁通过审核、谁分配客户"都要留痕——需要一张 audit_logs 数据库表，记录操作人、操作类型、资源、变更前后值。
6. 看板三层指标（内容层→获客层→转化层）
文档要求升级为三层指标。当前 dashboard_service.py 主要是内容层数据。获客层（各平台线索数/各账号线索数/加微率）和转化层（A/B/C/D 占比/首次响应时长/成交转化率）未实现。
7. 一键初始化脚本
文档要求填 .env → 执行一个命令 → 自动起服务 → 自动初始化库 → 自动创建管理员。当前有 docker-compose.yml 和 entrypoint.sh，但没有一个统一的 init.sh 脚本完成全部初始化。
