# API 路由治理规范

本规范用于约束后端路由演进，避免旧路径长期残留，并让前后端对版本边界有一致认知。

## 1. 路由版本策略

- 新增采集、素材、AI 工作台等演进中模块，优先进入版本化路由。
- 当前允许并存的路由层级：
  - `/api/v1/*`：第一代收口模块
  - `/api/v2/*`：结构已升级、返回模型已变化的模块
  - `/api/*`：历史稳定模块或尚未迁移模块
- 非紧急情况不再新增新的 `/api/*` 非版本化业务前缀。
- 新增接口前，必须先判断属于：
  - 现有稳定历史域
  - v1 已治理域
  - v2 已升级域
- 已迁移模块禁止保留双写逻辑。

## 2. 退役策略

- 旧路径一旦完成前后端切换，进入兼容期，而不是无限保留。
- 兼容层最长保留周期不超过 2 个迭代。
- 退役后必须补充以下验证：
  - 新路径接口可用
  - 旧路径返回 404
  - 前端无旧路径跳转

## 3. 当前状态（2026-04-01）

- 已进入 `/api/v1/*` 的模块：
  - `/api/v1/collect/*`
  - `/api/v1/ai/*`
  - `/api/v1/material/inbox/*`
  - `/api/v1/employee-submissions/*`

- 已进入 `/api/v2/*` 的模块：
  - `/api/v2/collect/*`
  - `/api/v2/materials/*`

- 仍保留历史非版本路由的模块：
  - `/api/auth/*`
  - `/api/customer/*`
  - `/api/lead/*`
  - `/api/publish/*`
  - `/api/dashboard/*`
  - `/api/system/*`
  - `/api/mvp/*`
  - `/api/insight/*`
  - `/api/topic/*`
  - `/api/traffic/*`

- 结论：
  当前项目处于“版本化收口中”，不是“全量 v1 化”。

## 4. 开发约束

- 新增端点必须在 PR 描述中给出：
  - 路由前缀
  - 依赖域服务
  - 回归测试清单
- 前端新增 API 调用必须优先复用统一路径常量，避免散落硬编码。
- 测试至少包含：
  - 1 个新路径 200 场景
  - 1 个旧路径 404 场景（若存在迁移历史）

## 5. 前端配套约束

- 新增前端接口优先放在 `desktop/src/api/*`。
- 路径常量统一定义在 `desktop/src/api/routes.ts`。
- `desktop/src/lib/api.ts` 视为历史聚合层，只做渐进迁移，不继续无节制扩张。
