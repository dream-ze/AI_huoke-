## 1. 正常流（Happy Path）

覆盖目标：核心业务正向链路端到端可通。

### 1.1 插件采集正向入库

- 用例：`test_v1_plugin_collect_syncs_to_content_and_insight`
- 操作：通过 `/api/v1/ai/plugin/collect` 提交平台、标题、正文、作者、标签、URL、热度
- 断言：
  - 返回 200
  - 响应含 `synced_content_asset_id`（已写入素材库）
  - 响应含 `synced_insight_item_id`（已写入洞察库）
  - `/api/content/list` 可查到标题匹配记录
  - `/api/insight/list` 可查到标题匹配记录
- 结果：**通过**

### 1.2 收件箱分拣→入素材库

- 用例：`test_v1_inbox_promote_workflow`
- 操作：创建 inbox → 分析 → 入库
- 断言：
  - inbox 创建返回 200
  - 分析后 status = `analyzed`
  - 入库后返回 `content_asset_id` + `insight_item_id`
  - inbox list 中该条状态变为 `imported`
- 结果：**通过**

### 1.3 统一采集入口正向投递

- 用例：`test_v1_collect_intake_to_inbox_with_dedupe_hint`（正向部分）
- 操作：`POST /api/v1/collect/intake`，来源 `mobile_share`
- 断言：首条投递返回 200，`dedupe_hit` = false
- 结果：**通过**

### 1.4 发布任务全生命周期

- 用例：`test_publish_task_lifecycle`
- 操作：创建 → 领取 → 回填 → trace → 查详情 → 驳回 → 关闭 → 统计
- 断言：
  - 创建后 status = `pending`
  - 领取后 status = `claimed`
  - 回填后 status = `submitted`，`post_url` 吻合
  - trace 返回 `lead_id` 非空
  - 详情含 feedbacks >= 3
  - 驳回后 status = `rejected`
  - 关闭后 status = `closed`
  - 统计 total >= 1，closed >= 1
- 结果：**通过**

### 1.5 线索运营全流转

- 用例：`test_lead_status_and_assign`
- 操作：创建任务 → 回填 → 获取线索 → 状态更新 → 归属 → 转客户 → 验证 list 含 customer_id → trace 含 customer_id
- 断言（摘要）：
  - lead list 含 publish_task_id 匹配记录
  - 状态更新为 `qualified`
  - 归属后 `owner_id` 非空
  - 转客户后 nickname 吻合
  - list 刷新后 `customer_id` 非空
  - lead trace 返回 `customer_id` 非空
- 结果：**通过**

### 1.6 客户 CSV 导出

- 用例：`test_export_customers_csv`
- 操作：先创建客户 → `GET /api/customer/export/csv`
- 断言：200，content-type = text/csv，含 nickname 列与数据
- 结果：**通过**

### 1.7 发布任务 CSV 导出

- 用例：`test_export_publish_tasks_csv`
- 操作：先创建任务 → `GET /api/publish/tasks/export/csv`
- 断言：200，content-type = text/csv，含 task_title 列与数据
- 结果：**通过**

### 1.8 企业微信通知触达

- 用例：`test_wecom_notify_requires_webhook`
- 操作：`POST /api/wecom/notify/test`
- 断言：未配置 webhook 时返回 400 或 502（符合预期防护）
- 结果：**通过**

## 2. 异常流（Error Path）

覆盖目标：非法输入、参数校验、边界错误能正确拦截并返回有意义错误码。

### 2.1 采集入口非法 URL

- 用例：`test_v1_collect_parse_link_invalid_url`
- 操作：`POST /api/v1/collect/parse-link`，url = `"invalid-url"`
- 断言：返回 400
- 结果：**通过**

### 2.2 采集入口非法来源类型

- 用例：`test_v1_collect_intake_invalid_source_type`
- 操作：`POST /api/v1/collect/intake`，source_type = `"unknown_source"`
- 断言：返回 422（参数校验失败）
- 结果：**通过**

## 3. 权限流（Permission Path）

覆盖目标：401/403 拦截机制生效，角色权限控制正确。

### 3.1 采集入口未授权拦截

- 用例：`test_v1_collect_intake_requires_auth`
- 操作：不携带 token 调用 `/api/v1/collect/intake`
- 断言：返回 401 或 403
- 结果：**通过**

### 3.2 线索转客户跨用户禁止

- 用例：`test_lead_convert_customer_forbidden_for_non_owner`
- 操作：用户 A 创建线索 → 用户 B 尝试转客户
- 断言：返回 403
- 结果：**通过**

### 3.3 Viewer 角色导出禁止

- 用例：`test_viewer_role_cannot_export`
- 操作：将用户角色设为 `viewer` → 请求 CSV 导出
- 断言：客户导出返回 403，发布任务导出返回 403
- 结果：**通过**

## 4. 幂等/去重流（Idempotency & Dedup Path）

覆盖目标：重复数据提交不污染系统、去重机制生效。

### 4.1 统一采集入口去重检测

- 用例：`test_v1_collect_intake_to_inbox_with_dedupe_hint`（去重部分）
- 操作：
  1. 首次投递 URL = `https://example.com/post/abc?utm=1`（mobile_share） → 成功，`dedupe_hit` = false
  2. 再次投递相同基础 URL（不同 utm 参数、不同来源 wechat_forward） → 成功，`dedupe_hit` = true，`duplicate_ids` >= 1
- 断言：系统基于 URL 规范化去重，跨来源可识别重复
- 结果：**通过**

### 4.2 收件箱去重预览

- 用例：`test_v1_inbox_batch_actions_and_dedupe_preview`（去重预览部分——注：批量指派部分因参数校验 422 失败，详见第 6 节）
- 操作：创建 2 条相同 source_url 的 inbox 记录 → `GET /api/v1/inbox/dedupe/preview`
- 断言：`total_duplicates` >= 2
- 现状：去重预览逻辑通过；批量 assign 因入参 schema 变动导致 422，已登记修复项
