# 回滚手册（发布失败应急）

更新时间：2026-03-23

## 1. 触发条件

- 新版本健康检查失败（`/api/v1/health` 不可用）
- 核心链路不可用（登录、发布任务、线索转客户、插件采集）
- 错误率持续升高且无法在 15 分钟内止血

## 2. 快速回滚流程

1. 确认回滚指令与负责人。
2. 记录当前故障版本（镜像 tag / commit hash）。
3. 停止当前容器：

```bash
docker compose -f backend/docker-compose.yml down
```

4. 切换到上个稳定版本并启动：

```bash
git checkout <stable_tag_or_commit>
docker compose -f backend/docker-compose.yml up -d
```

5. 运行健康检查与关键链路抽样。
6. 在 30 分钟内补齐问题记录与后续修复计划。

## 3. 数据回滚原则

- 默认先回滚应用，不直接回滚数据库。
- 仅在迁移明确可逆且经过评估时执行 `alembic downgrade -1`。
- 涉及数据丢失风险时，先做备份再执行回退。

## 4. 回滚后检查

- [ ] API 健康检查恢复
- [ ] 登录与鉴权可用
- [ ] 发布任务 -> 线索 -> 客户链路可用
- [ ] 插件回传接口可用
- [ ] 日志与监控恢复正常

## 5. 记录要求

- 必填文档：`docs/operations/问题记录模板.md`
- 必填信息：故障时间线、影响范围、根因、修复与预防项
